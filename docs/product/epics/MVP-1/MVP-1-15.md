### S15: Subagent Session Ingestion

**As Mustafa**, I want every subagent invocation (developer, tester, tech-lead,
architect, etc.) to be ingested as its own session, linked to its parent session via
`parent_session`, so that knowledge produced by a specific agent type is queryable in
isolation and agent-scoped views work.

#### Context

Each Claude Code session directory contains a `subagents/` subdirectory with
per-subagent pairs:

```
<project>/<parent-session-uuid>/subagents/
├── agent-<aid>.meta.json   # {"agentType":"tech-lead","description":"<task summary>"}
└── agent-<aid>.jsonl       # full subagent transcript (isSidechain:true, agentId)
```

Across full Claude Code history this is **~417 subagent sessions** with type breakdown:
developer (112), Explore (91), tester (59), tech-lead (51), architect (32),
general-purpose (31), product-manager (19), react-ui-designer (8), Plan (7), and others.
All of these are silently skipped today because `ingest.py` only scans top-level
`*.jsonl` files.

The subagent `.jsonl` first user-record contains the **full context prompt** the
subagent received — this is extremely valuable for understanding what task the agent
was actually asked to do, not just the truncated `description` in `.meta.json`.

#### Acceptance criteria

- [-] `ingest.py` discovers `<project>/<session-uuid>/subagents/agent-*.jsonl` files
  in addition to top-level `*.jsonl` files.
- [-] For each discovered subagent JSONL, the sibling `.meta.json` is read and:
  - `agent_type` (from `agentType`) populates the manifest `agent` field
  - `agent_task` (from `description`) is written to a new manifest field
  - `parent_session` is set to the outer session UUID (the parent directory name)
- [-] Each subagent becomes its own manifest entry with its own `session_id`
  (derived from the subagent JSONL filename, e.g. `agent-a7b3a2427ff7af6b7`).
- [-] The subagent bronze copy is written to `bronze/` with a filename that keeps the
  subagent distinguishable: `YYYY-MM-DD_<parent-session>_<agent-id>.jsonl` (date from
  `session_started_at` via S14 logic).
- [-] `manifest.py` schema comment block documents the new `agent_task` field and
  clarifies the semantics of `parent_session` for subagents.
- [-] `compact.py` silver frontmatter propagates `agent_task` and `parent_session`.
- [-] `extract.py` passes agent type + agent task to the extraction prompt so gold
  entries can cite the invoking agent's perspective (e.g. "during a tech-lead review,
  a subagent identified ...").
- [-] Idempotency: re-running `ingest.py` does not re-copy subagent files already in
  the manifest.
- [-] Unit tests cover:
  - Subagent discovery in a nested directory
  - `.meta.json` parsing (present, missing, malformed)
  - Parent-session linkage
  - Idempotency across a re-run

#### Acceptance criteria — Refinement (added 2026-04-25)

The original implementation lifted `agentType` and `description` from `.meta.json`
into the manifest, but did not copy the `.meta.json` file itself into bronze. This
breaks the bronze self-containment principle: any downstream stage running remotely
(Claude Routine, cloud) cannot reach `~/.claude/projects/`, so any field we did not
explicitly extract is lost forever after Claude Code's 30-day source GC. Plus, the
flat `bronze/` directory mixes data from one harness only today — adding harness
namespacing now is cheap insurance for future Codex/Hermes/etc. sources.

- [-] **Bronze is harness-namespaced**: the bronze directory layout becomes
  `bronze/<harness>/<file>` (defaulting to `bronze/claude-code/` since `harness`
  field already serializes as `"claude-code"`). `DEFAULT_BRONZE_DIR` parent stays
  `bronze/` but every per-session write goes into the harness subdir.
- [-] **Subagent `.meta.json` is copied to bronze** alongside its sibling JSONL.
  The bronze layout for a subagent becomes:
  - `bronze/claude-code/YYYY-MM-DD_<parent>_<agent-id>.jsonl`
  - `bronze/claude-code/YYYY-MM-DD_<parent>_<agent-id>.meta.json`
  Both files are byte-identical copies of the source. If `.meta.json` is missing
  in the source (rare — claude-mem auto-generated subagents), nothing is copied
  for that file (the JSONL still ingests as before).
- [-] **New manifest field**: `meta_path` (string or null) is the absolute path to
  the bronze copy of `.meta.json`, or null for top-level sessions and subagents
  whose source had no `.meta.json`. Documented in `manifest.py` schema comment block.
- [-] **Migration**: existing bronze files are relocated to `bronze/claude-code/`
  and all manifest entries' `bronze_path` strings are updated to point at the new
  location. Migration is a one-shot operation done as part of this story; document
  the commands used in story comments. Bronze is gitignored, so the move has no
  repo impact.
- [-] **arc42 gets a "Bronze Self-Containment" subsection** under §Bronze Layer
  stating the principle: bronze is the immutable, complete archive of source-side
  artifacts; the manifest is a derived index for fast filtering, never the sole
  source of provenance metadata. Future stages must be able to run with only
  `bronze/` + `manifest.jsonl` accessible.
- [-] **Tests**:
  - `.meta.json` copied alongside subagent JSONL
  - `meta_path` populated correctly in manifest (subagent with meta, subagent
    without meta, top-level session)
  - Bronze paths in fresh ingest contain `claude-code/` segment
  - Idempotent re-ingest does not re-copy `.meta.json`

### Migration commands (2026-04-25)

Bronze directory (`bronze/`) is gitignored and was not present on disk at the time
of this refinement (confirmed by `ls bronze/` returning ENOENT). Therefore no file
`mv` was needed. Only the manifest required migration.

```bash
# Step 1: Back up manifest before surgery.
cp manifest.jsonl manifest.jsonl.bak-2026-04-25

# Step 2: Update all bronze_path values and add meta_path: null.
# (Python script used instead of jq due to field insertion requirement.)
python3 - <<'EOF'
import json
from pathlib import Path

manifest_path = Path('manifest.jsonl')
with open(manifest_path) as f:
    entries = [json.loads(l) for l in f if l.strip()]

for entry in entries:
    bp = entry.get('bronze_path')
    if bp and '/bronze/' in bp and '/bronze/claude-code/' not in bp:
        entry['bronze_path'] = bp.replace('/bronze/', '/bronze/claude-code/')
    if 'meta_path' not in entry:
        entry['meta_path'] = None

with open(manifest_path, 'w') as f:
    for entry in entries:
        f.write(json.dumps(entry, ensure_ascii=False) + '\n')
EOF

# Step 3 (if bronze dir exists with files to move):
# mkdir -p bronze/claude-code
# mv bronze/*.jsonl bronze/claude-code/
# (Skipped: bronze/ was not present on disk -- gitignored, not yet populated locally.)
```

Verification:
- `jq -c 'select(.bronze_path != null) | .bronze_path' manifest.jsonl | grep -v "/bronze/claude-code/" | head`
  returns nothing (all paths migrated).
- `jq -c 'select(.parent_session != null) | {session_id, meta_path}' manifest.jsonl | head`
  shows subagent entries with `meta_path: null` (none had source .meta.json at time
  of original ingest; future ingest runs will populate meta_path for new subagents).

#### Architect Notes

This is the first story that introduces **many-to-one** relationships in the manifest
(one parent → N subagents). The graph is implicit via `parent_session`; no new
"relationships" table is needed.

Watch for size: subagent JSONLs vary from 10 records to a few hundred. Most are well
under the `LARGE_SESSION_BYTES_THRESHOLD` (500 KB) and will compact fine. Large
architect/developer subagent runs occasionally approach the limit; that's acceptable —
they fall into `skipped-large` like any other session.

Boundary: do NOT collapse a subagent into its parent's silver. They are separate
sessions with separate intents. Cross-linking happens at the silver frontmatter / gold
level via `parent_session`.
