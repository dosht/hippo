### MVP-2-01: MVP-1 Bug Fixes

[Test Plan](test-plans/MVP-2-01.md)

**As a pipeline operator**, I want the nightly pipeline to recover gracefully from quota walls,
ignore plugin-spawned trash sessions, track stable project identity, respect a configurable history
cutoff, and consolidate multi-day work into one gold entry per thread, so that a real nightly run
produces correct, non-fragmented gold without manual intervention.

This story covers five issues. Issue 4 (throughput) is explicitly closed without action (see
Technical Notes below).

---

#### Issue 1: Quota wall permanently marks sessions failed

**Problem.** `compact.py` raises `QuotaExhaustedError` and exits cleanly. `scripts/extract.py` and
`scripts/reconcile.py` do not. A quota wall mid-run leaves those sessions with status `failed`,
requiring manual manifest surgery to recover.

**Solution.** Port the `QuotaExhaustedError` raise-and-exit pattern from `compact.py` to
`scripts/extract.py` and `scripts/reconcile.py`. On quota wall: log the wall, do not update
the in-flight session status (it remains silver for extract, or unchanged for reconcile), and
exit with code 0 consistent with the arc42 Section 8 "Quota-Aware Graceful Stop" pattern so
launchd does not flag the job as failed.

**Acceptance criteria.**

- [x] `extract.py` catches quota errors and exits without writing `failed` to the manifest for
  in-flight sessions. Session status reverts to `silver`.
- [x] `reconcile.py` catches quota errors and exits without writing `failed` to the manifest.
  In-flight cluster status reverts to `gold-candidate` or `silver` as appropriate.
- [x] The 1,588 rows in `manifest.jsonl` that were mismarked `failed` due to quota are confirmed
  reset to `bronze`. Verification: `jq '[.[] | select(.status=="bronze")] | length' < manifest.jsonl`
  count matches the expected post-reset value. The backup at `manifest.jsonl.bak.20260428-143535`
  is the reference.
- [x] A simulated quota wall in the test suite (mock `QuotaExhaustedError` raised mid-run) leaves
  no new `failed` rows and exits with the documented retry code.

---

#### Issue 2: claude-mem and plugin-spawned sessions ingested as real sessions

**Problem.** Sessions spawned by claude-mem, agent orchestrators, or other plugins appear in
`~/.claude/projects/` alongside real user sessions. They carry no signal worth ingesting and pollute
the gold corpus.

**Solution.** Add a `SessionStart` Claude Code hook (`scripts/hooks/session_start.py`). On every
session start, the hook writes a sidecar record to `~/.hippo/sessions.jsonl`:

```
{"session_id": "<id>", "project_id": "<sha1>", "branch": "<branch>", "thread_id": "<sha1>", "started_at": "<iso8601>"}
```

The ingest script requires a matching sidecar entry to admit a JSONL into bronze. Plugin-spawned
sessions (claude-mem, subagents spawned outside the global hook) do not fire the global
`SessionStart` hook, so they produce no sidecar record and are never ingested.

**Acceptance criteria.**

- [x] `scripts/hooks/session_start.py` is the only new file introduced by this story.
  Bronze/silver/gold/QMD/hippo-remember skill are unchanged.
- [x] `ingest.py` rejects any JSONL that has no corresponding entry in `~/.hippo/sessions.jsonl`.
  Rejected sessions are logged at INFO level but not written to the manifest.
  - Note: the sidecar gate is opt-in via `--sidecar PATH`; when not provided the gate is disabled for backward compat. See deviation note in tech-lead review below.
- [x] A test fixture with a mock JSONL that has no sidecar entry is rejected by ingest.
- [x] A test fixture with a mock JSONL that has a matching sidecar entry is admitted.

---

#### Issue 3: Unreliable project identity

**Problem.** Project identity is currently resolved at ingest time, when the branch may already
be deleted, worktrees may have moved, or symlinks may have shifted. The result is inconsistent
`project_id` across sessions that belong to the same project.

**Solution.** The `session_start.py` hook (Issue 2) resolves `project_id` at session start, while
the git context is still intact:

- `project_id = sha1(git remote get-url origin)` if a remote exists
- `project_id = sha1("local:" + repo_root_abspath)` if no remote but git repo present
- `project_id = null` if no git context

The sidecar record carries `project_id`. Ingest reads `project_id` from the sidecar and writes it
to the manifest row, overriding any previously-inferred value. The manifest schema gains
`project_id` and `sidecar_path` as new fields (backward-compatible; absent on pre-MVP-2 rows).

**Acceptance criteria.**

- [x] `session_start.py` resolves `project_id` using the three-case logic above and writes it to
  the sidecar record.
- [x] `ingest.py` reads `project_id` from the sidecar and populates it in the manifest row.
- [x] Sessions from the same git remote in different worktrees produce the same `project_id`.
- [x] Sessions with no git context produce `project_id: null` in the manifest.

#### Issue 4: Throughput too slow (CLOSED WITHOUT ACTION)

This issue was listed in the MVP-2 planning notes and is explicitly closed here without any
implementation. Quota is the binding constraint on pipeline throughput, not parallelism or
per-session processing speed. Adding parallelism before fixing the quota recovery path
(Issues 1-3) would complicate the codebase without reducing wall-clock time.

Any future throughput work should be reopened as a separate story after the quota recovery
path is validated in production.

#### Issue 5: History flood (pre-cutoff sessions ingested on first run)

**Problem.** On first run against a large `~/.claude/projects/` corpus, all historical sessions
are eligible for ingest. This causes a massive backlog that cannot complete before the quota wall
and produces an unmanageable backlog in the manifest.

**Solution.** Add `HIPPO_INGEST_FROM=YYYY-MM-DD` config flag. Ingest reads this from
`~/.hippo/config` (INI or YAML, operator-managed) or from the environment variable directly.
Sessions whose `started_at` (from the sidecar) predates the cutoff are permanently skipped at
ingest time. They are never written to the manifest. No backfill path.

Default cutoff: `2026-04-28` (the day the MVP-1 quota incident occurred).

**Acceptance criteria.**

- [x] `ingest.py` reads `HIPPO_INGEST_FROM` from `~/.hippo/config` and then from the environment,
  with environment taking precedence.
- [x] Sessions whose sidecar `started_at` is before the cutoff are skipped and logged at DEBUG
  level. They are never written to the manifest.
- [x] Sessions with no sidecar (Issue 2 gate) are rejected before the cutoff check, so the
  cutoff logic is not reached for unsidecarred sessions.
- [x] Default cutoff is `2026-04-28` when `HIPPO_INGEST_FROM` is absent from both config and env.
- [x] A test verifies that a session with `started_at: 2026-04-27` is skipped under the default
  cutoff, and a session with `started_at: 2026-04-28` is admitted.

#### Issue 6: Multi-day work fragmented across sessions

**Problem.** A developer works on the same feature across three days, producing three separate
sessions. The reconcile cluster step treats them as unrelated, producing three small gold entries
instead of one consolidated entry covering the full arc of work.

**Solution.** The `session_start.py` hook computes:

```
thread_id = sha1(project_id + ":" + branch)
```

The hook writes `thread_id` to the sidecar. Compact writes `thread_id` into the silver frontmatter.
Extract carries `thread_id` through to the gold frontmatter. The reconcile cluster step adds
`thread_id` to its cluster keys alongside the existing topic-similarity clustering, so sessions
sharing a `thread_id` are always co-clustered regardless of content similarity.

Reused branch names collapse to the same thread. This is an accepted trade-off (see Technical
Notes).

**Acceptance criteria.**

- [x] `session_start.py` computes `thread_id = sha1(project_id + ":" + branch)` and writes it to
  the sidecar.
- [x] `compact.py` reads `thread_id` from the manifest row (populated by ingest from the sidecar)
  and writes it to the silver YAML frontmatter.
- [x] `extract.py` carries `thread_id` from silver frontmatter to gold frontmatter.
- [x] `reconcile.py` cluster step includes `thread_id` as a cluster key. Sessions sharing a
  `thread_id` are always placed in the same cluster.
- [x] A test fixture with three sessions sharing the same `project_id` and branch produces a
  single cluster, not three.

#### Technical Notes

**Sidecar as the single new file.** `scripts/hooks/session_start.py` is the only new file
introduced by this story. All other changes are modifications to existing scripts.

**Issue 4 closed without action.** Throughput optimization is deferred. The note in the story body
above is the permanent record; do not reopen in MVP-2.

**Reused branch names.** `thread_id = sha1(project_id + ":" + branch)` means a branch named
`main` used across two unrelated features produces the same `thread_id`. This is acceptable per
design: in practice, feature work happens on feature branches; `main` commits are typically small
and consolidation is harmless.

**Manifest backup.** `manifest.jsonl.bak.20260428-143535` was created before the quota-reset
operation that cleared 1,588 `failed` rows back to `bronze`. Issue 1 acceptance criteria requires
verifying that reset was applied correctly before any new pipeline run in this story.

**New manifest fields.** `project_id`, `sidecar_path`, and `thread_id` are new fields on manifest
rows. They are absent on pre-MVP-2 rows (backward-compatible). Scripts must not crash on rows that
lack these fields.

#### Dependencies

None. This is the foundation story for MVP-2.

#### Architectural decisions

**Exit code 0 on quota wall.** All three quota-aware scripts (compact.py, extract.py, reconcile.py)
exit 0 on `QuotaExhaustedError`, consistent with arc42 §8 "Quota-Aware Graceful Stop" and the
existing compact.py line 679 pattern. This prevents launchd from flagging the job as failed (the
launchd plists have no `SuccessfulExit` key). The nightly runner distinguishes a quota stop from
a successful run via the manifest state (no new gold entries written), not the exit code.

**`QuotaExhaustedError` in `scripts/errors.py`.** The class is defined once in `scripts/errors.py`
and imported by compact.py, extract.py, and reconcile.py. This is the single source of truth for
the detection heuristic (stderr pattern matching). `scripts/__init__.py` already exists, so no
new module initialisation is needed.

**Sidecar index loaded once at ingest startup.** This is safe because ingest is a short-lived
process. The sidecar file is append-only and not modified during a run. No locking needed.

**No config file parser dependency.** `~/.hippo/config` is read with a simple line-by-line INI
parser (stdlib `configparser`) to avoid adding a new dependency.

**thread_id in reconcile cluster step.** The union-find DSU already handles arbitrary merge
operations. Adding a thread_id pass before writing clusters requires only reading gold frontmatter
(already done by `parse_entry`) and calling `dsu.union()` for thread-grouped pairs. No new data
structure needed.

### Tech Lead Review (2026-04-29)

**Status: approved_with_hints**

The plan is well-scoped and internally consistent. All issues have clear AC. The task breakdown maps cleanly to individual functions. Notes below are coaching hints, not blockers.

**Exit code 75 vs exit 0 (Tasks 1, 2)**

`pipeline.md` (Graceful quota stop pattern) documents `return 0` so launchd does not flag the job as failed. The plan specifies exit code 75. These must be reconciled before implementation. Check whether the launchd plist has `SuccessfulExit` configured. If not, exit 75 will appear as a job failure in `launchctl list`. Either update the plists or use exit 0 as the guide says. The architectural note at the bottom of the plan acknowledges the nightly runner needs updating, but that runner update is out of scope, so there is a risk of shipping exit 75 to a runner that treats it as an alert.

**`QuotaExhaustedError` duplication (Tasks 1, 2)**

The plan copies the class definition into `extract.py` and `reconcile.py`. `pipeline.md` shows the canonical definition. These should live in one shared location. `scripts/__init__.py` already exists. Either put it there or create `scripts/errors.py`. Duplicating the class across three files means three places to update when the classification logic changes (CODE_REVIEW.md item 3 calls out exactly this).

**Task 2.4 -- QMD quota classification**

`run_cluster` calls `qmd_query` (a local CLI, not the Claude API). QMD does not return HTTP 429. Classifying a QMD failure as `QuotaExhaustedError` conflates two unrelated error modes. If QMD fails, it is a local indexing issue, not a quota wall, and exit 75 would cause the nightly runner to silently skip clustering indefinitely. This task should be removed or changed to mark a `failed`-with-error and exit non-zero distinctly.

**Backward-compat on manifest reads (Task 4.3)**

The plan says scripts must tolerate missing keys with `.get()`. Verify that `compact.py` and `extract.py` already use `.get("thread_id")` and `.get("project_id")` before adding code that writes these fields, otherwise you may mask a `KeyError` with silent `None` propagation. Confirm with a grep before starting Task 6.

**Testing standards**

`testing.md` says there is no test suite yet and lists `tests/fixtures/sample_session.jsonl` as the fixture path. Task 7 adds tests to `test_extract.py`, `test_reconcile.py`, etc. Make sure the test files are created under `tests/` (not `scripts/tests/`) and that fixtures go to `tests/fixtures/` per the guide. Also per the guide: do not mock the `claude -p` subprocess directly; inject a fake callable instead (Task 7.1 and 7.2 mock `QuotaExhaustedError`, which is correct, but verify the injection point is the wrapper function, not the subprocess itself).

**`load_ingest_cutoff` return type (Task 5.1)**

The signature `load_ingest_cutoff() -> str` returns a date string. Downstream comparison (`sidecar["started_at"][:10] >= cutoff`) is a lexicographic string compare, which is correct for ISO8601 dates. Add a type hint and a brief comment documenting this invariant so the next reader does not replace it with a datetime comparison unnecessarily.

**Commit scope**

Each task maps to a distinct module. Commit them separately with the scopes defined in `coding-standards.md` (`ingest`, `compact`, `extract`, `hooks`). Avoid a single omnibus commit for all seven tasks.
