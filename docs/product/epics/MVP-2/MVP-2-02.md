### MVP-2-02: Optimization Phase 1

**As a pipeline operator**, I want each session to progress from silver to gold-candidate before the
next session begins, and I want the static prompt prefix to be cached by the Anthropic API, so that
a quota-walled run still produces partial gold output and so that per-call token costs drop
measurably on repeated runs.

---

#### Item 1: Per-session inlined compact + extract

**Problem.** The current pipeline runs all compactions across all sessions, then all extractions,
then reconcile. Under quota pressure, compact never finishes for all sessions, so extract never
runs, so no gold is produced at all from a quota-walled run. The operator gets nothing.

**Solution.** Change the main pipeline loop to process each session end-to-end before moving to the
next:

```
for each eligible session:
    compact(session)   -> silver
    extract(session)   -> gold candidate (written to gold/entries/ immediately)
    on QuotaExhaustedError: stop loop, leave remaining sessions in bronze
reconcile()            -> run at end of loop, or separately on a weekly schedule
```

On `QuotaExhaustedError` from either compact or extract: stop the entire loop immediately. Sessions
processed before the wall have gold candidates on disk. Sessions not yet reached remain in bronze
(not failed). The operator can resume the loop on the next quota refresh.

Reconcile remains batch. It runs at the end of the full loop (all sessions complete) or as a
separate weekly scheduled run. It is not inlined per-session.

**Acceptance criteria.**

- [ ] After a quota-walled run (simulated via mock `QuotaExhaustedError` raised at session N),
  sessions 1..N-1 have gold candidate files in `gold/entries/`. Session N and beyond remain in
  bronze.
- [ ] No session is left with status `failed` due to the quota wall (consistent with MVP-2-01
  Issue 1 fix).
- [ ] Reconcile is not called until the loop completes or is invoked as a standalone command.
- [ ] The pipeline loop is interruptible: a `SIGINT` during processing stops the loop at the
  current session boundary (after the current extract finishes or after the current compact
  finishes), not mid-file.

---

#### Item 2: Prompt caching for compact and extract

**Problem.** The compaction prompt (`scripts/prompts/compact.md`) and the extraction prompt
(`scripts/prompts/extract.md`) are static across all calls in a run. The Anthropic API bills the
full prompt prefix on every call. For a run with 50 sessions, the same 2-4K token prompt is billed
50 times.

**Solution.** Use Anthropic prompt caching (`cache_control: {"type": "ephemeral"}` on the static
message block) at the call sites in `compact.py` and `extract.py`. The static prompt text becomes
the cached prefix. The per-session bronze/silver content is appended as the uncached suffix. On
cache hit, the prefix tokens are billed at the cached rate (approximately 10% of normal).

This is a single API change per call site. No changes to the prompt files themselves. No changes to
the pipeline orchestration.

**Acceptance criteria.**

- [ ] `compact.py` sends the static prompt block with `cache_control: {"type": "ephemeral"}` on
  the first message in the API call.
- [ ] `extract.py` sends the static prompt block with `cache_control: {"type": "ephemeral"}` on
  the first message in the API call.
- [ ] The Anthropic API response includes `cache_read_input_tokens > 0` on the second and
  subsequent calls in the same run (confirmed via logged usage stats).
- [ ] Total `cache_read_input_tokens` across a 5-session test run exceeds `cache_creation_input_tokens`,
  confirming the cache is being hit and not just created.
- [ ] No change in silver or gold output quality between a cached and uncached run on the same
  input (output determinism is acceptable within the model's normal variance).

---

#### Technical Notes

**Prompt caching TTL.** The Anthropic prompt cache has a 5-minute TTL. For a run that takes longer
than 5 minutes between calls to the same prompt, cache hits are not guaranteed. This is acceptable:
the optimization is best-effort and falls back to normal billing without breaking correctness.

**No changes to bronze/silver/gold/QMD/hippo-remember.** This story modifies only the pipeline
orchestration loop and the Anthropic API call sites in `compact.py` and `extract.py`.

**Reconcile scheduling.** The per-session loop produces gold candidates, not finalized gold. The
reconcile step (cluster + judge) remains the gate for promotion to stable gold. Operators who want
fresh gold quickly can run `python -m scripts.reconcile` immediately after the loop completes.
A weekly cron schedule is the recommended default.

#### Dependencies

Depends on MVP-2-01. The sidecar gate (Issue 2) and quota recovery pattern (Issue 1) must be in
place before the per-session loop is safe to run in production. Without the sidecar gate, the loop
would process plugin-spawned sessions. Without the quota recovery fix, a loop interruption would
still leave failed rows.
