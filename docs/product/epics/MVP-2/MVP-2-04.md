### MVP-2-04: Speed Up hippo-remember Skill

**As an agent user**, I want a hippo-remember query to complete in 10-15 seconds instead of ~47
seconds, so that reaching for memory feels like a natural reflex rather than a costly detour.

This is a research and benchmarking story. No production code lands here. The output is a
benchmarked comparison of at least three speedup options against the current baseline, plus a
written recommendation for the implementation story that follows.

---

#### Background: Current Latency Breakdown

The approximate breakdown of the current 47-second round trip:

- Outer `claude` cold start (Node + MCP init + auth): ~5s
- Outer model decides to spawn skill, runs Bash: ~2s
- Inner `claude -p` cold start (the subagent): ~5s
- QMD query (embed + BM25 + rerank): ~3s
- Read top-N entries from disk: ~1s
- Inner Haiku synthesis (~200 words): ~10-15s
- Outer model formats final reply: ~10-15s

The dominant cost is two reasoning steps plus two `claude` cold starts. Subagent context
isolation must be preserved. Pollution risk is real and was an explicit user decision. Any
speedup option must not collapse the inner subagent back into the outer agent's context.

**Hard constraint (project policy).** Direct Anthropic API calls are off the table. Every option
must run through either the Claude Code subscription or a local-model alternative.

---

#### Options to Evaluate

Each option below should be benchmarked for wall-clock gain and assessed for answer-quality
impact.

**Option 1: Trim MCP load in the inner `claude -p`.**

Pass `--strict-mcp-config` with an empty or minimal MCP config so the inner subagent skips MCP
discovery and handshake. The inner subagent only needs Bash and Read. Removing MCP init from the
inner cold start is expected to save 2-4 seconds.

- Pros: zero code change to the skill logic, applies immediately.
- Cons: requires the operator to maintain a separate minimal MCP config file; brittle if the
  inner skill ever needs an MCP tool.
- Expected gain: 2-4s.

**Option 2: Prompt caching for the inner system prompt.**

The 600-token append-system-prompt is identical on every hippo-remember invocation. Ensure
`cache_control` is set so subsequent queries reuse the prefix. The Claude Code subscription
supports prompt caching; the cache TTL is 5 minutes. Repeated queries within a session benefit;
cold first-query of the day does not.

- Pros: no structural change, purely additive.
- Cons: only helps repeated queries within a short window; no gain on the first query of a
  session.
- Expected gain: 3-6s on cache-hit invocations.

**Option 3: Skip QMD rerank when top BM25 score is high (above 0.85).**

The nightly log already short-circuits query expansion with "Strong BM25 signal -- skipping
expansion." Generalise that pattern to the rerank step too. When the top-1 BM25 score exceeds
0.85, skip the reranker and return the BM25 result directly.

- Pros: targeted, low risk of quality regression when the shortcut fires.
- Cons: marginal gain, fires only on easy queries.
- Expected gain: 1-3s when shortcut fires.

**Option 4: QMD daemon mode.**

Keep the QMD process warm so the embedding model does not reload on each invocation. Expected to
eliminate model load time on the QMD side.

- Pros: simple operational change, no code.
- Cons: adds a background process to manage; gain is marginal.
- Expected gain: 0.5-1s.

**Option 5: Pi coding agent with a local model as the inner subagent.**

Replace the inner `claude -p` subagent with the Pi coding agent
(https://github.com/badlogic/pi-mono, https://mariozechner.at/posts/2025-11-30-pi-coding-agent/)
running a local model via ollama (candidates: Gemma 3, Qwen 2.5 7B, Llama 3.1 8B). Pi has a
4-tool core (read, write, edit, bash) with no MCP overhead and a dramatically lighter cold start.
A local model avoids Anthropic API latency entirely for the synthesis step. Subagent context
isolation is preserved by the separate process. For context on free CLI agents as a category,
see https://parallel.ai/blog/free-CLI-agent.

- Pros: highest-leverage option; eliminates both the inner `claude` cold start and the Haiku
  synthesis latency; no API cost for the inner step.
- Cons: most unknowns; local model quality on faithful synthesis is unproven; two LLM stacks
  to maintain (Claude subscription for the heavy pipeline, local model for retrieval-summarize);
  Pi skill syntax may not map cleanly to the existing hippo-remember skill without porting work.
- Expected gain: brings total round trip to ~10-15s vs. 47s (rough estimate pending benchmark).

This is the highest-leverage option and the primary research target.

---

#### Open Questions

The following are research targets for this story, not assumptions to be made upfront:

1. **Quality of small local models on synthesis.** A faithful echo of gold entry content matters.
   Hallucination defeats the entire point of the memory system. An A/B test design is needed:
   same query on Haiku-via-claude vs. Gemma 3 vs. Qwen 2.5 7B, evaluated against gold-entry
   ground truth for factual accuracy (port numbers, exact error messages, concrete file paths).

2. **Pi skill compatibility.** Does Pi's tool and skill syntax map cleanly to the existing
   hippo-remember skill structure? If not, what is the porting cost? Can the inner skill run
   unmodified or does it need a Pi-specific variant?

3. **Operational burden of two stacks.** Claude subscription for the heavy nightly pipeline,
   local model for retrieval-summarize. Is the speed gain worth maintaining a second inference
   stack? Quantify the gain first, then decide.

4. **Staged adoption path.** Should retrieval-summarize stay on Haiku-via-claude with MCP
   trimmed (Option 1), and only fall back to Pi+ollama if Option 1 does not move the needle
   enough? A two-phase rollout may reduce risk.

#### Acceptance Criteria

This is a draft research story. Picking it up means:

- [ ] Wall-clock benchmarks recorded for the current baseline (47s, broken down by phase).
- [ ] At least three options from the list above measured against the baseline on the same set
  of five representative queries.
- [ ] Answer quality assessed A/B: at least Haiku-via-claude vs. one local model (Gemma 3 or
  Qwen 2.5 7B), evaluated against gold-entry ground truth on factual recall of concrete values.
- [ ] A written recommendation (1-2 paragraphs) naming the preferred path and rationale,
  suitable as the input spec for the implementation story that follows.
- [ ] No production code changes land as part of MVP-2-04. Benchmark scripts may be committed
  under `scripts/bench/` but are not part of the pipeline.

---

#### Technical Notes

**No direct API calls.** The project policy is explicit: all LLM calls go through the Claude Code
subscription or a local-model alternative. Do not introduce `anthropic` SDK calls or raw HTTP
calls to api.anthropic.com.

**Subagent isolation is non-negotiable.** The inner subagent runs in its own context by design.
Options that collapse the inner step into the outer agent's context are out of scope regardless
of speed gain.

**QMD stays.** QMD is not replaced in this story. Retrieval layer changes are out of scope for
MVP-2 (see epic non-goals). The speedup search is limited to cold-start costs, caching, and the
inner inference stack.

#### Dependencies

None. This story can be picked up independently of MVP-2-01 through MVP-2-03.
