### MVP-2-05: Automatic Agent Invocation of hippo-remember

**As an agent user**, I want developer, tester, and tech-lead agents to consult hippo-remember
automatically when starting a task, hitting an error, or facing a non-trivial decision, so that
the memory corpus improves agent behavior without requiring me to type `/hippo-remember` every
time.

This is an exploration story. The output is a design document that picks one or two mechanisms,
plus a small experiment plan with measurable success criteria. No production-grade implementation
lands in this story.

---

#### Why This Matters

The "zero-effort write, intelligent read" principle from CLAUDE.md is half-met today. The write
side is automatic via the SessionStart hook and the nightly pipeline. The read side requires
explicit user invocation. That makes hippo a reference tool, not a memory system.

The research priors in `docs/brainstorm/related-work.md` are relevant here. MemRL (near-miss
failures drive recall), ERL (transferable heuristics), and Hindsight's recall operation all
assume passive recall, not on-demand lookup. Until reads are automatic, the corpus value is
invisible to the agent during real work.

The user's concrete framing: "I want it to be as frequent as read or bash tools."

---

#### Mechanisms to Evaluate

**Mechanism 1: System-prompt nudge.**

Add a section in CLAUDE.md (or in per-agent definitions) instructing the agent to call
hippo-remember when: (a) starting a story or new task, (b) hitting a tool error, (c) about to
make a decision in a domain that has past gold entries.

- Pros: zero code, ships today.
- Cons: instructions compete with other context under pressure; agents may ignore them when the
  context window is large or the task is urgent.

**Mechanism 2: Subagent default reflex.**

Update each agent definition (`team:developer`, `team:tester`, `team:tech-lead`) to include a
"before you start, ask hippo-remember about: [keywords derived from the story]" preamble. Stronger
than Mechanism 1 because the instruction is co-located with the task description rather than
buried in a global config.

- Pros: per-agent, task-scoped, hard to miss.
- Cons: adds boilerplate to every agent definition; keywords must be manually or automatically
  derived.

**Mechanism 3: Hook-driven auto-recall (UserPromptSubmit equivalent).**

A Claude Code hook that fires on user prompt submit, runs `qmd query` against the prompt text,
and surfaces top-K entries inline as a `<hippo-context>` block in the system prompt. A bounded
version (3 entries max, 200 chars per entry body, only fires when top-1 relevance score is above
a threshold) may avoid the context-pollution problem that caused the v2 detour to be rejected.

- Pros: fully automatic, no agent definition changes needed.
- Cons: context pollution risk is real and was the explicit reason this approach was rejected
  before; bounded version may be too restrictive to be useful; requires careful threshold tuning.
  Only revisit if a bounded design demonstrably avoids pollution.

**Mechanism 4: Bash tool wrappers that pre-query memory.**

Before a Bash call that matches a known-failure pattern (e.g., lemonsqueezy webhook, pgtap,
supabase SSR), inject the relevant memory entry inline. Tightly targeted, low pollution, but
requires maintaining a pattern registry and only helps on known-failure domains.

- Pros: surgical precision, near-zero false-positive rate for known patterns.
- Cons: medium implementation cost; only helps in domains already represented in the gold corpus;
  pattern registry needs ongoing maintenance.

**Mechanism 5: Trigger on inner monologue markers.**

When the agent self-narrates phrases like "let me try", "I think", or "I'm not sure", inject a
hippo-remember query for that moment. Behavioral, not architectural; no hook or wrapper needed.
Requires prompt engineering to make the agent surface these markers reliably.

- Pros: no infrastructure, targets the exact moments of uncertainty.
- Cons: hard to test reliably; depends on model verbosity in chain-of-thought.

---

#### Recommendation (Record Before Picking Up)

Start with Mechanisms 1 and 2. They are the cheapest options and can ship today without code.
Measure invocation rate and false-positive rate over one week of real use. Use that data to
decide whether Mechanisms 3, 4, or 5 are worth implementing.

This story is explicitly tied to MVP-2-04. Automatic invocation only makes sense if a single
query is fast enough to be amortized over a real task. Adding 47 seconds of latency every time
the agent reaches for memory makes the feature counterproductive. Do not pick up MVP-2-05 until
MVP-2-04 has concluded with a latency recommendation.

---

#### Open Questions

The following are research targets, not assumptions:

1. **Cost model.** Each automatic invocation costs 10-47s (depending on MVP-2-04 outcome). At
   what frequency does that cost exceed acceptable latency? An agent working on a 2-hour story
   might invoke hippo-remember 20-50 times. Even at 10s per query, that is 3-8 minutes of
   compounded latency. What is the user's tolerance threshold?

2. **False-positive rate.** What fraction of automatic queries return entries that are actually
   relevant? If agents act on irrelevant entries, behavior degrades rather than improves. "Not
   Always Faithful" cuts both ways: agents follow raw concrete experiences, so misapplied entries
   cause concrete misbehavior, not just noise.

3. **Discovery vs. confirmation.** Is the automatic call meant to discover relevant past
   knowledge (broad query, surface anything semi-related) or confirm a specific concern (narrow
   query, only fire when stuck)? The two modes have different precision and recall trade-offs and
   may require different triggering conditions.

4. **Interaction with the Reflect pattern.** Hindsight calls for synthesizing across memories.
   If 10 agents each call hippo-remember 30 times per session, do we accumulate a useful audit
   log of which entries were consulted? Can that feed into reconcile to strengthen high-recall
   entries? This connects MVP-2-05 back to MemRL strength scoring as a future story.

5. **Permission friction.** The skill spawns a memory subagent that requires user approval on
   first invocation. For automatic invocation this is unworkable. Either approval needs to be
   one-time-and-remembered, or the skill needs to run inline without spawning a separate process.
   This is a prerequisite for any automatic mechanism, not a detail.

#### Technical Notes

**Context pollution is the primary risk.** The v2 detour was rejected specifically because
injecting memory into the outer agent's context caused unpredictable downstream behavior. Any
automatic mechanism must have a clear upper bound on the context it injects (entry count, body
length, relevance threshold).

**QMD stays.** Retrieval layer changes are out of scope for MVP-2. Automatic invocation uses the
existing `qmd query` CLI exactly as hippo-remember does today.

**Agent definitions are in `.claude/agents/`.** Mechanism 1 and Mechanism 2 patches target files
in that directory. Changes there do not require a pipeline deployment.
