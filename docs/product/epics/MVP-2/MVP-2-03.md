### MVP-2-03: Optimization Phase 2

**As a pipeline operator**, I want small sessions routed to a cheaper model, reconcile to skip
clusters that did not change, and the eligibility-filter question permanently closed, so that the
recurring nightly run cost drops on a stable corpus without sacrificing gold quality.

---

#### Item 1: Route by session size (model routing)

**Problem.** All sessions use `claude-sonnet-4-5` (or current Sonnet) regardless of size. A
5 KB bronze session costs nearly as much as a 500 KB session at the prompt level. Small sessions
contain less nuance; Haiku is sufficient for compaction and extraction at that size.

**Solution.** Add a routing threshold, configurable via `HIPPO_HAIKU_THRESHOLD_KB` (default: 100).
Sessions whose bronze file size is below the threshold use `claude-haiku-*` (latest available
Haiku) for both compact and extract. Sessions at or above the threshold use Sonnet.

Escalation path: if the reconcile judge marks a Haiku-processed silver as low quality (confidence
below a configurable floor, default 0.5), the session is re-queued for Sonnet compaction. The
manifest gains a `compact_model` field recording which model was used.

**Acceptance criteria.**

- [ ] `compact.py` and `extract.py` read `HIPPO_HAIKU_THRESHOLD_KB` and select the model
  accordingly.
- [ ] `manifest.jsonl` rows gain a `compact_model` field (`haiku` or `sonnet`).
- [ ] A test fixture with a 50 KB bronze file uses Haiku; a 200 KB bronze file uses Sonnet.
- [ ] When the reconcile judge assigns confidence below the floor to a Haiku-compacted session,
  the session is re-queued as `bronze` with a `retry_with_sonnet: true` flag in the manifest.
- [ ] A manual review of 5 Haiku-compacted silvers (from a real run) confirms acceptable quality
  before this story is closed. "Acceptable" means the silver preserves at least one near-miss
  and at least one concrete value (path, port, error message) per MemRL and "Not Always Faithful"
  criteria.

---

#### Item 2: Incremental reconcile

**Problem.** `reconcile cluster --full` re-judges every cluster on every nightly run. On a stable
corpus where most clusters have not received new candidates since the last run, this re-judgment
is wasted quota.

**Solution.** Add an incremental mode (default for nightly runs; full mode retained as a manual
override). Incremental mode:

1. Reads `last_reconciled_at` per cluster from the manifest (new field, absent on pre-MVP-2 rows
   defaults to epoch).
2. Identifies clusters that received at least one new gold candidate since `last_reconciled_at`.
3. Re-clusters and re-judges only those clusters.
4. Updates `last_reconciled_at` for each re-judged cluster.

Full mode (`reconcile cluster --full`) re-judges every cluster regardless of `last_reconciled_at`.
Recommended as a weekly safety net.

**Acceptance criteria.**

- [ ] `reconcile.py` defaults to incremental mode. `--full` flag forces full re-judgment.
- [ ] A run on a corpus where no new candidates arrived since the last reconcile produces zero
  judge API calls (all clusters are skipped). Confirmed via logged stats.
- [ ] A run where 3 of 20 clusters received new candidates re-judges exactly those 3 clusters.
- [ ] `last_reconciled_at` is written to the manifest after each incremental run.
- [ ] Full-mode output is identical to incremental-mode output on a corpus where all clusters
  changed (regression test).

---

#### Item 3: Eligibility filter (CLOSED WITHOUT ACTION)

A previous suggestion proposed filtering out "pure tool" sessions (sessions with no assistant
turns, only tool calls) before they enter the pipeline, on the grounds that they carry no
explanatory content.

**This is explicitly rejected.** Tool-only sessions carry near-miss, procedural, and environmental
learning that is valuable per MemRL (Jan 2026): near-miss failures are more valuable than clean
successes, and tool-failure sequences are exactly the kind of near-miss that the pipeline should
capture. Hermes (Nous Research, 2025-2026) similarly retains all session types in its four-layer
memory, not just sessions with rich assistant narration.

This decision is recorded here permanently so it is not reopened in MVP-3 or later. If a future
operator finds that a specific class of tool-only sessions is producing low-quality gold, the
correct fix is to improve the compaction prompt for that class, not to gate them out at ingest.

**Acceptance criteria.**

- [ ] No eligibility filter by session type exists in `ingest.py`, `compact.py`, or `extract.py`
  after this story is merged.
- [ ] This story file (MVP-2-03.md) is the authoritative record of the rejection decision. It is
  not removed or amended without a new story that explicitly revisits and supersedes it.

#### Technical Notes

**Model routing thresholds are tunable.** `HIPPO_HAIKU_THRESHOLD_KB` is an env var or config key.
The default of 100 KB is a starting point; operators should adjust based on observed quality after
the first few production runs.

**Incremental reconcile and thread_id.** The `thread_id` cluster key introduced in MVP-2-01 is
already factored into the incremental reconcile logic. A cluster is considered "changed" if any of
its member sessions have `last_updated_at` after the cluster's `last_reconciled_at`, regardless of
whether they share a `thread_id`.

**No changes to gold frontmatter schema.** `compact_model` and `last_reconciled_at` are manifest
fields only. Gold frontmatter is unchanged.

**Quality regression gate.** The reconcile judge step already produces a confidence score per gold
entry. The acceptance criteria for this story require no drop in that score distribution across a
before/after comparison on the same corpus. The developer should run a before/after on a 20-session
sample and record the score distributions before closing this story.

#### Dependencies

Depends on MVP-2-02. Model routing requires the per-session pipeline loop from MVP-2-02 to be in
place so that routing decisions and escalation re-queuing can happen at session granularity rather
than at batch level.
